# Phase 3 — Records Export

**Goal:** One click produces all impairment records for a property in a date range. This directly answers the packet's central crisis.
**Time:** 1.5 hours
**Depends on:** Phase 1 (more data to export makes it meaningful)

---

## Why This Matters

Robert Halberd (Day +4): "I want every inspection report for Cedar Heights for the past three years on a thumb drive by end of business Friday."
Marshal Reyes: wants 3 years of ITM records within 30 days.
Worth Patel: 19 specific documents under preservation hold.

If Beacon had ImpairmentOS, Tom could respond in under a minute.

---

## Tasks

### 3.1 — Backend: Export endpoint
- [ ] New file: `backend/app/routes/export.py`
- [ ] `GET /properties/{id}/export?start_date=YYYY-MM-DD&end_date=YYYY-MM-DD`
- [ ] Query all impairments for the property within date range
- [ ] For each impairment, reuse the packet generation logic from `packets.py`
  - Extract `generate_packet_data(db, impairment)` as a helper function in `backend/app/services/packet_service.py`
  - Import into both `packets.py` and `export.py`
- [ ] Return shape:
  ```json
  {
    "property": { "name", "address", "owner_entity", "property_manager", "carrier_name", "carrier_account" },
    "jurisdiction": { "name", "nfpa25_edition", "local_code_ref" },
    "date_range": { "start": "2023-01-01", "end": "2026-05-08" },
    "summary": {
      "total_impairments": 3,
      "closed": 2,
      "open": 1,
      "closed_incomplete": 0,
      "avg_duration_hours": 3.8,
      "compliance_rate": 0.67
    },
    "impairments": [
      { "packet": { ...same shape as /impairments/{id}/packet... } }
    ]
  }
  ```
- [ ] Register router in `backend/main.py`

### 3.2 — Backend: Extract packet generation helper
- [ ] New file: `backend/app/services/packet_service.py`
- [ ] Move the body of `get_packet()` in `packets.py` into `generate_packet_data(db, imp)` function
- [ ] Refactor `packets.py` to call the helper
- [ ] Import helper in `export.py`

### 3.3 — Frontend: Add export types and API call
- [ ] In `frontend/src/api.ts`:
  - Add `ExportData` type matching the response shape above
  - Add `api.getExport(propertyId, startDate, endDate)` method

### 3.4 — Frontend: Export modal component
- [ ] New file: `frontend/src/components/ExportModal.tsx`
- [ ] Triggered by "Export Records" button (add to NavBar or Dashboard header)
- [ ] Simple modal with:
  - Property dropdown (required, populated from `api.getProperties()`)
  - Start date input (default: 3 years ago)
  - End date input (default: today)
  - "Generate Export" button
- [ ] On submit: navigate to export view with the selected params

### 3.5 — Frontend: Records export view
- [ ] New file: `frontend/src/components/RecordsExport.tsx`
- [ ] Fetches from the export endpoint and renders as a structured document:
  - Header: "IMPAIRMENT RECORDS — [Property Name]"
  - Subtitle: "Date range: [start] to [end]"
  - Property details (owner, PM, carrier, jurisdiction)
  - Summary stats table
  - Each impairment rendered as a condensed packet:
    - Impairment number, status, duration
    - Timeline (condensed — just key events with timestamps)
    - Compliance checklist (pass/fail per item)
  - Footer: "Generated by ImpairmentOS — [timestamp]"
- [ ] "Back to Dashboard" and "Print" buttons (no-print class)

### 3.6 — Wire into App.tsx
- [ ] Add view type: `{ type: "export"; propertyId: number; startDate: string; endDate: string }`
- [ ] Add routing in App.tsx render logic
- [ ] NavBar or Dashboard: "Export Records" button opens the modal
